# Dependency Resolution

TorcPy supports two types of job dependencies: **explicit** (declared in the spec) and
**implicit** (inferred from file and user-data relationships).

## Explicit Dependencies

Declare dependencies directly in the workflow spec using job names:

```yaml
jobs:
  - name: step_a
    command: echo A

  - name: step_b
    command: echo B
    depends_on: [step_a]   # step_b waits for step_a

  - name: step_c
    command: echo C
    depends_on: [step_a, step_b]   # step_c waits for both
```

### Regex Dependencies

You can also match dependencies by regex pattern. This is useful for fan-in after
parameterized jobs:

```yaml
jobs:
  - name: train_{i}
    command: python train.py --fold {i}
    parameters:
      i: "1:5"

  - name: aggregate
    command: python aggregate.py
    depends_on_regexes:
      - "train_.*"     # waits for all 5 train jobs
```

## Implicit Dependencies (via Files)

When job A writes a file and job B reads that same file, TorcPy automatically creates a
dependency: B depends on A.

```yaml
files:
  - name: raw_data
    path: data/input.csv
    st_mtime: 1710000000.0   # pre-existing input — no job writes this

  - name: features
    path: data/features.csv   # generated by preprocess

jobs:
  - name: preprocess
    command: python preprocess.py
    input_files: [raw_data]
    output_files: [features]

  - name: train
    command: python train.py
    input_files: [features]   # implicit dep on preprocess (writes features)
```

The dependency graph becomes:

```mermaid
flowchart LR
    pre[preprocess] --> train[train]
    style pre fill:#3f51b5,color:#fff
    style train fill:#2e7d32,color:#fff
```

### The `st_mtime` Field

The `st_mtime` field on a `FileSpec` distinguishes input files from output files:

- **Set** (`st_mtime: <timestamp>`) → pre-existing input file. No job needs to produce it.
- **Not set** (`st_mtime: null`) → output file. Some job must write it.

Only files with no `st_mtime` participate in implicit dependency creation.

## Implicit Dependencies (via User Data)

The same logic applies to `user_data` items:

```yaml
user_data:
  - name: config
    data: {learning_rate: 0.001}

  - name: best_params
    is_ephemeral: false

jobs:
  - name: tune
    command: python tune.py
    input_user_data: [config]
    output_user_data: [best_params]

  - name: train_final
    command: python train_final.py
    input_user_data: [best_params]   # implicit dep on tune
```

## Initialization

Dependencies are built when you call `initialize`:

```console
torcpy workflows initialize <id>
```

Or automatically when running:

```console
torcpy run <spec_or_id>
torcpy workflows run <id>
```

During initialization:

1. Implicit dependencies are inserted into `job_depends_on` (from file/user_data relationships)
2. Jobs with no dependencies → `ready (2)`
3. Jobs with unmet dependencies → `blocked (1)`

## Circular Dependency Detection

!!! warning
    TorcPy does not currently detect circular dependencies at initialization time. A circular
    dependency will result in all involved jobs remaining `blocked` indefinitely. Ensure your
    workflow graph is a DAG (directed acyclic graph).

## Background Unblocking

When a job completes, TorcPy does **not** immediately check and unblock dependents. Instead:

1. The `complete_job` endpoint sets `unblocking_processed = 0` on the completed job.
2. A background `asyncio.Task` runs on a 1-second interval.
3. The background task finds all completed jobs with `unblocking_processed = 0`.
4. For each, it checks whether all dependencies of each blocked dependent are now complete.
5. Satisfied dependents are transitioned to `ready`.

This design ensures job completion endpoints remain fast even in large workflows.
